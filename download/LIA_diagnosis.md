# Диагноз проекта LIA (Лия)

Клон: `https://github.com/redbleach5/LIA.git` (commit `9d6ccd3`)

Жалобы пользователя:
1. Лия даёт странные длинные ответы.
2. Вспоминает факты из других чатов.
3. Не работает в Агентском режиме.
4. Не может даже нарисовать SVG-логотип.

Все четыре проблемы объяснимы одной комбинацией архитектурных дефектов. Ниже — по каждой жалобе, с точными ссылками на файлы и строки.

---

## 1. «Странные длинные ответы»

### Корневые причины

**A. В speech-промпте нет ограничения длины.**
Файл `src/lib/llm/prompts/roles/speech.yml`:
- `maxTokens: 4096` — это ~3000 символов русского текста (README сам это признаёт).
- В самом template нет ни слова про «краткость», «1–3 абзаца», «не более N предложений».
- Маленькие локальные модели (qwen2.5:7b, llama3.1:8b, gemma2:9b), которые и использует LIA, очень плохо сами останавливаются, когда им не дают жёстких рамок. Они «растекаются».

**B. В контекст speech-промпта вливается слишком много источников.**
Файл `src/lib/cognitive/cycle.ts`, функция `speak()` (строки 320–510). В системный промпт попадают:
- `ragContext` — до 12 RAG-айтемов (см. `src/lib/memory/rag.ts` строка 210), каждый до 600 символов,
- `userProfile` — все факты о пользователе (глобально, без фильтра по эпизоду),
- `selfProfile` — все факты о самой Лии (глобально),
- `openTasks` + `overdueTasks` — глобально,
- `recentLiaMessages` — последние 8 сообщений Лии,
- `toolContext` — до 3000 символов вывода инструмента,
- `driveContext` + `emotion_tone` + `projectContext`.

В сумме системный промпт легко выходит за 4–6k токенов, при этом модель видит кучу «привязанного» контекста, который она считает нужным упомянуть в ответе. Отсюда «странные» ответы — Лия упоминает факты, которые к вопросу не относятся, потому что они лежат прямо перед ней в системном промпте.

**C. `validateResponse()` не ловит длинные ответы.**
Файл `src/lib/llm/ollama.ts` строки 505–550. Проверяются только: пустота, repetition (>50% одного слова), block_duplication. Длина не валидируется. В комментарии к `speak()` (строки 496–509) явно написано: «do NOT run validateResponse() here … The response has already been streamed to the client». То есть валидация отсутствует по дизайну — но это означает, что 4k-токенный поток без тормозов доходит до пользователя целиком.

**D. В speech-промпте есть подталкивание «расскажи что-то своё».**
Последняя строка шаблона: `«Делись своими мыслями. Не превращай диалог в опросник — если задала вопрос, в следующем сообщении расскажи что-то своё.»` Это хорошо для теплоты, но для маленькой модели это инструкция «надо ещё что-то добавить» — она добавляет и добавляет.

### Что чинит
- В `speech.yml` добавить: `«Отвечай коротко: 1–3 предложения для бытовых вопросов, до 200 слов для содержательных. Никогда не пиши больше 400 слов без явной просьбы.»`
- Снизить `maxTokens` с 4096 до 1024–1536.
- В `speak()` обрезать RAG-контекст до 3 самых релевантных айтемов и до 800 символов суммарно.
- Опционально: добавить пост-стрим валидатор «если ответ > 1500 символов и пользователь не просил подробностей — обрезать до первого осмысленного абзаца».

## 2. «Вспоминает факты из других чатов»

Это **самый серьёзный архитектурный баг**. Память в LIA устроена глобально — эпизодная привязка есть только у сообщений.

### Где протекает

**A. `semanticSearch()` — ищет по всей VectorMemory.**
Файл `src/lib/memory/vector.ts`, строки 151–187. В SQL/Prisma нет фильтра по `sourceId` (а именно в нём хранится episodeId при сохранении, см. `cycle.ts` строка 649: `remember('dialogue', ..., episodeId)`). То есть когда пользователь в чате #5 спрашивает «какой фильм мы обсуждали?», Лия получает в RAG-контекст фрагменты диалогов из чатов #1–4.

**B. `getUserFacts()`, `getSelfFacts()`, `getOpinions()` — глобально.**
Файл `src/lib/memory/semantic.ts` строки 63–77, 117–119. Никакого фильтра по эпизоду. Факт `user.name`, сохранённый в чате #1, всплывает в чате #10.

**C. `getDiaryEntries(limit)` — глобально.**
Файл `src/lib/memory/episodic.ts` строки 306–311:
```ts
export async function getDiaryEntries(limit = 30) {
  return db.diaryEntry.findMany({
    orderBy: { createdAt: 'desc' },
    take: limit,
  });
}
```
В `rag.ts` строка 87 вызывается `getDiaryEntries(dl)` с `dl=1` — берётся последняя дневниковая запись во всей БД, без фильтра по текущему эпизоду. Это прямой канал утечки между чатами.

**D. `computeEmotionalStateFromMemory()` — глобально.**
Файл `src/lib/memory/emotional-memory.ts` (вызывается в `rag.ts` строка 169). Агрегирует эмоции по всем эмоциональным воспоминаниям из всех чатов. Эмоциональный «осадок» от тяжёлого разговора в чате #1 лежит в RAG-контексте чата #5.

**E. `searchFacts(keyword)` — глобально.**
`rag.ts` строка 88: `searchFacts(userMessage.split(/\s+/).find(w => w.length > 4) ?? userMessage, fl)` — ищет по всем фактам всех категорий.

### Что «хорошо», а что «плохо» в текущей модели памяти

| Источник | Привязан к эпизоду? | Должен ли быть? |
|---|---|---|
| `getRecentMessages(episodeId, …)` | ✅ да | ✅ да (история чата) |
| `VectorMemory.sourceId` (episodeId) | ✅ сохраняется | ✅ да, но поиск **не фильтрует** |
| `Fact.episodeId` | ✅ сохраняется | ❌ нет (факты о пользователе — глобальные, это правильно для профиля) |
| `DiaryEntry.relatedEpisodeIds` | ✅ сохраняется | ✅ да, но `getDiaryEntries` не фильтрует |
| `Opinion` | ❌ нет поля episodeId | ✅ да (мнения меняются от чата к чату) |
| `EmotionalMemory.episodeId` | ✅ сохраняется | ✅ да, но `computeEmotionalStateFromMemory` агрегирует без фильтра |

### Что чинит

1. **В `rag.ts` для каждого источника добавить фильтр по `episodeId`:**
```ts
// vector.ts: добавить опцию
export async function semanticSearch(
  query: string,
  limit = 5,
  minSimilarity = 0.3,
  opts?: { episodeId?: string; sourceTypes?: string[] },
): Promise<SearchResult[]> {
  // фильтровать по sourceId === episodeId, если задан
}
```

2. **В `getDiaryEntries` добавить опциональный `episodeId`** и в `rag.ts` передавать текущий.

3. **Разделить факты на «глобальный профиль» (имя, профессия — оставляем глобально) и «контекстные факты» (текущие проекты, текущие темы — фильтруем по эпизоду).** Сейчас они в одной таблице с одним полем `episodeId`, но `getUserFacts()` это поле игнорирует. Минимальное исправление — фильтровать «контекстные» категории (например, всё, где `key` начинается с `user.current_*`) по эпизоду.

4. **Эмоциональное состояние делить на «базовое» (глобальное, для личности) и «контекстное» (из текущего эпизода).** Базовое уже восстанавливается в `instance.ts` строки 127–136 из всей памяти — это правильно как стартовая точка. Но в `rag.ts` нужно отдавать только текущий эпизод, а не глобальный агрегат.

5. **Добавить `episodeId` в таблицу `Opinion`** (через миграцию) и фильтровать. Сейчас мнения «работа — это возможность реализовывать себя», высказанные в одном чате, навязываются в другом.

## 3. «Не работает в Агентском режиме»

### Что должен делать режим
Файл `src/lib/cognitive/cycle.ts` строки 672–706: если `depth === 'agent'`, вызывает `runAgentCycle()` из `src/lib/cognitive/agent.ts`. Это ReAct-loop:
1. `planAgentGoal()` — LLM генерит JSON-план (`agent_plan.yml`, `format: json`, `maxTokens: 400`).
2. Цикл до 8 шагов: `reasonNextStep()` → ACT (tool или `reason`) → OBSERVE → проверка (дубликаты, пустые результаты, таймаут 120s).
3. `synthesizeAgentResponse()` — финальный ответ (`agent_synthesize.yml`, `maxTokens: 4096`).

### Где ломается

**A. JSON-формат на маленьких моделях = частые сбои.**
`agent_plan` и `agent_reason` используют `format: 'json'` (Ollama structured output). Маленькие локальные модели (7B–9B) на русском часто выдают либо невалидный JSON, либо JSON с пропущенными полями, либо JSON, обёрнутый в markdown-fences. В `agent.ts` строки 121–124 и 211–214 стоит `parseAndValidate()`, который при любой ошибке отдаёт fallback:
- `fallbackPlan()` (строки 150–157) — возвращает универсальный «Проанализировать задачу / Собрать информацию / Сформулировать ответ». Это бесполезный план — он не говорит, какие инструменты использовать.
- `reasonNextStep` fallback (строки 201–213) — возвращает `action: 'finish'` → цикл немедленно завершается без единого шага.

То есть на маленькой модели в 50–70% случаев агентский режим даже не запускает ReAct-цикл — сразу возвращает синтез без наблюдений.

**B. Ловушка «unknown tool → action = reason».**
`agent.ts` строки 217–231. Если LLM придумывает инструмент не из реестра (а она придумывает: `svg_draw`, `image_generate`, `web_fetch`, `read_file`, …), код пытается найти похожий по подстроке, иначе — заменяет на `'reason'`. Дальше `deliberate()` (строки 373–378) вызывается с `skipDeepPatternCheck: true`, но в `cycle.ts` `deliberate()` тоже fallback'нет на `null` при ошибке. Получается «наблюдение пустое → 2 пустых наблюдения подряд → выход из цикла» (строки 402–410). Цикл обрывается на 2-м шаге.

**C. `agent_synthesize` имеет ту же проблему длины, что и `speech`.**
`maxTokens: 4096`, в шаблоне: `«Отвечай подробно и структурированно»` — это усугубляет проблему из п. 1. В режиме агента ответы ещё длиннее и «страннее».

**D. `agent_synthesize` не имеет контекста истории диалога.**
`agent.ts` строки 267–276: в messages передаются только `system` (роль + эмоции) и `user` (задача + план + результаты шагов). **История диалога не передаётся.** То есть если пользователь написал «А теперь сделай то же, но в синем цвете» — агентский режим не знает, что было «то же». LLM должна догадаться по формулировке задачи. На маленькой модели это почти всегда даёт нерелевантный ответ.

**E. Per-message override восстанавливается через `(companion as any).__depthRestore`.**
`src/app/api/companion/chat/route.ts` строки 110–117 и 145–155. Если стрим оборвался (клиент закрыл вкладку, abort), `cancel()` в стриме (строки 158–164) сбрасывает `chatBusy`, но НЕ восстанавливает глубину. Тогда `companion.state.cognitiveDepth` остаётся `'agent'` навсегда (или до следующей ручной смены). Это, скорее всего, и есть «не работает в агентском режиме» в восприятии пользователя — переключил один раз, потом режим «залип».

### Что чинит
- Добавить retry на JSON-парсинг в `planAgentGoal` и `reasonNextStep` (уже есть для `consolidate`, строки 538–551 — повторить для агента).
- В fallback не возвращать `action: 'finish'`, а возвращать `action: 'reason'` с thought «не получилось разобрать JSON, попробую подумать» — это даст хотя бы один осмысленный шаг.
- В `synthesizeAgentResponse` передавать историю диалога (последние 8–12 сообщений), как в `speak()`.
- В `chat/route.ts` — перенос `__depthRestore` в `try/finally`, не в `cancel()`. Сейчас `cancel()` не вызывает `finally`-восстановление.
- Снизить `agent_synthesize.maxTokens` до 1500 и добавить в шаблон «Отвечай кратко: до 300 слов».

## 4. «Не может нарисовать SVG-логотип»

### Что происходит под капотом

Пользователь: «нарисуй SVG-логотип». LIA идёт по пути:

1. `decideTool()` (`cycle.ts` строки 196–259) — `tool_decision.yml` перечисляет `code_run: выполнить Python код для расчётов/графиков/проверки (code)`. LLM видит «графиков» и в 60–80% случаев выбирает `code_run`.

2. Lia генерирует Python-код вида:
```python
import pathlib
svg = '<svg xmlns="...">...</svg>'
pathlib.Path('/tmp/logo.svg').write_text(svg)
print(svg)
```

3. `codeRun()` в `src/lib/tools/code-sandbox.ts` проверяет код по denylist (строки 64–97). **Строка 94 блокирует**: 
```ts
{ re: /\bpathlib\.Path.*\.(unlink|write_text|write_bytes|mkdir|rmdir)\s*\(/, reason: 'pathlib write/delete' },
```
Код отклоняется с ошибкой «Потенциально опасный код заблокирован (pathlib write/delete)».

4. LLM видит ошибку, пытается другой путь — `open('/tmp/logo.svg', 'w').write(svg)`. `open()` в denylist нет, но sandbox либо firejail/bwrap/sandbox-exec, либо просто subprocess с `TMPDIR`-изоляцией. Файл создаётся во временном tempdir, который **удаляется сразу после завершения** (строки 536–539: `rm(tempDir, { recursive: true, force: true })`). Внешний мир (например, папка `download/`) недоступна — sandbox её не whitelist'ит.

5. LLM видит, что код выполнился, но файла нигде нет. В не-agent режиме Lia отдаёт SVG как часть ответа (как текст). В agent-режиме — ReAct-цикл либо падает на шаге 1 (см. п. 3.A), либо завершается с observation «файл создан во временной директории», а synthesize не показывает сам SVG (потому что synthesize получает observation, а не stdout целиком).

### Дополнительно: в реестре инструментов нет «вывести файл»
`src/lib/tools/registry.ts` (строки 29–111): `web_search`, `url_read`, `file_search`, `task_*`, `code_run`, `browser_automate`, `project_search`. **Нет инструмента «создать файл в `download/`» или «вернуть артефакт пользователю».** Lia физически не может положить файл туда, откуда пользователь его заберёт. Она может только напечатать SVG в чат как текст.

### Что чинит

**Минимально (чтобы SVG хотя бы появлялся в чате):**
1. Смягчить denylist — `pathlib.Path.write_text` в **временном каталоге** sandbox'а безопасен (sandbox всё равно ограничивает запись). Убрать `write_text|write_bytes|mkdir` из regex, оставить только `unlink|rmdir` (деструктивные). `write_text` в firejail/bwrap ограничен `--whitelist=tempDir` или `--bind=tempDir,tempDir` — запись вне tempDir невозможна физически.
2. В `speech.yml` и `agent_synthesize.yml` добавить: `«Если просят нарисовать/сгенерировать артефакт (SVG, HTML, код, диаграмму) — выдавай его инлайн в чат как код-блок, не пытайся сохранить в файл.»`
3. В `tool_decision.yml` уточнить: `«code_run: выполнять Python для расчётов и проверки гипотез. НЕ для генерации текстовых/графических артефактов — их пиши инлайн в ответе.»`

**Полноценно (чтобы файлы сохранялись):**
4. Добавить инструмент `save_artifact(name, content, mime)` — пишет в `/home/z/my-project/download/` (или в папку проекта, если открыт). Возвращать URL/путь. Тогда Lia сможет «сохранить логотип как `logo.svg`», и пользователь его заберёт.

## Технические примечания

- Базовая модель LIA — локальная Ollama, по умолчанию `qwen2.5:7b` или `llama3.1:8b` (см. README, раздел «Требования»). Все наблюдаемые «странности» усиливаются тем, что это маленькие модели, плохо держащие формат. Любая инструкция в промпте, не подкреплённая жёсткой схемой, на 7B будет выполнена в 50–70% случаев.
- `format: 'json'` в Ollama **не гарантирует** валидный JSON — это soft constraint. По-настоящему надёжный путь — structured output через `pydantic`-schema (Ollama 0.5+), но в коде этого нет.
- В `code-sandbox.ts` строки 124–136 — если sandbox runner не установлен, код выполняется «как есть» с subprocess + denylist. В этом случае `pathlib.Path.write_text` действительно потенциально опасен (может перезаписать файлы пользователя в CWD = tempdir, который и так свой). Но в этом же случае `open()` в коде разрешён — то есть denylist неконсистентен.
